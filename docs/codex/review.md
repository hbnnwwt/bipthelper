# Codex 项目技术审计报告

## 1. 核心判断
**结论：** 这不是工程，这是原型。勉强能跑，但距离生产级代码差了十万八千里。
**理由：** 典型的"AI 快速拼凑"产物。堆砌了时髦的技术栈（FastAPI、MeiliSearch、LLM），但每一层都停留在"能跑就行"的层面，缺乏对边界条件、错误处理和系统稳定性的任何敬畏。

---

## 2. 架构层面批判

### 2.1 数据结构：简陋且不负责任
- **模型设计：** `Document` 模型仅有 5 个核心字段，`content` 字段无长度限制，直接把原始 HTML 炼成的文本塞进去，完全不考虑数据库膨胀和检索性能。
- **哈希索引：** `content_hash` 用 MD5？现在都 2026 年了，还在用这种已经被证伪的算法？更可笑的是，用它来判断"内容是否更新"，一旦发生哈希碰撞（虽然概率低，但在工程上不能接受），你的爬虫就会漏掉关键更新。
- **AI 状态：** `ai_status` 只有 4 个状态，用字符串存储？为什么不用 Enum？状态机的流转逻辑在哪？`pending` -> `failed` 后谁来重试？

### 2.2 爬虫逻辑：线程安全的笑话
- **锁的滥用：** `crawler.py` 第 25 行的 `_crawl_lock` 竟然只保护了一个布尔值 `crawl_running`？这种写法在并发环境下不仅低效，而且充满竞态条件。如果两个请求同时触发 `crawl_all`，第二个请求会直接被吞掉，连个错误都不报。
- **状态管理混乱：** 全局变量 `crawl_progress` 和 `crawl_stop_requested` 在多线程环境下，虽然用了 `threading.Lock()`，但你的锁粒度完全不对，只锁了"开始/结束"，中间的状态更新根本没保护。如果 API 线程在读 `crawl_progress` 时，爬虫线程正在写，你拿到的是脏数据。
- **错误处理：** 整个爬虫用了多少个 `try-except`？大部分异常被吞掉，只打个 `logger.error`，然后继续跑。这是在掩耳盗铃，错误会像滚雪球一样越积越多，直到系统崩溃你都不知道原因。

### 2.3 AI 集成：业余到令人发指
- **同步调用：** `categorize_article` 在爬虫主线程中同步调用 LLM，一旦 LLM 超时（你设置的 120 秒），整个爬虫就会卡住 2 分钟。你的调度器每分钟跑一次，结果前一次还没跑完，后一次就又开始了，系统资源会被瞬间耗尽。
- **Prompt 设计：** `categorize.py` 的 `DEFAULT_PROMPT_TEMPLATE` 竟然只传标题？文档分类的核心信息是内容，你把最有价值的信息扔了，只靠标题猜分类？这叫"智能分类"？这叫"瞎猜"。
- **解析逻辑：** `_parse_json_response` 用正则提取 JSON？LLM 返回的文本可能包含各种格式的"前后缀"，你的正则覆盖率有多少？测过吗？一旦解析失败，整个文档的 AI 状态就被标记为 `failed`，再也不会被处理。

## 3. 搜索层：从入门到放弃

### 3.1 MeiliSearch 的滥用
- **索引配置：** `update_filterable_attributes` 和 `update_sortable_attributes` 在每次获取索引时都调用一次？这是在干什么？索引配置应该在启动时做一次，或者在数据变更时做，你把它放在 `get_index()` 里，每次搜索都会触发一次更新请求，这是在自找性能瓶颈。
- **转义逻辑：** `_escape_filter` 只转义了双引号？MeiliSearch 的过滤语法还支持单引号、反斜杠、括号等特殊字符，你的转义完全不完整，用户输入一个包含特殊字符的关键词，你的搜索就会崩。
- **分页：** `estimatedTotalHits` 是一个估算值，在分页场景下极不稳定。用户点第 5 页可能就没有数据了，你的前端做好处理了吗？

---

## 4. API 层：毫无章法

### 4.1 职责不清
- **`ai.py` 第 27-78 行：** `list_providers` 接口竟然在 API 层处理 JSON 模板和数据库的合并逻辑？这是业务逻辑，为什么要写在 API 层？你的 `services` 层是摆设吗？
- **`save_provider`：** 第 80-132 行，60 行代码，全是重复的 `if-else` 和数据库操作。这种代码不仅难读，而且极易出错。为什么不用 ORM 的 `update` 方法？为什么不用 Pydantic 模型做数据验证？

### 4.2 安全性：形同虚设
- **CORS 配置：** `allow_origins=["http://localhost:3000"]`，这是开发环境的配置，生产环境呢？如果忘记改，你的 API 会暴露给所有人。
- **加密存储：** `encrypt_value` 和 `decrypt_value` 的实现呢？我看不到，但如果只是简单的 AES，没有密钥轮转，没有硬件安全模块（HSM），所谓的"加密"就是在自欺欺人。

## 5. 自然语言问答（RAG）功能审计

### 5.1 实现可能性：理论上可行，实际是灾难
当前系统具备 RAG 的基础组件（检索引擎 + LLM 调用），但如果直接在上面实现，只会制造出更多垃圾。

### 5.2 致命缺陷
1. **检索质量：** MeiliSearch 是关键词检索引擎，它不懂语义。用户问"如何处理含硫废气？"，它只会返回包含"含硫"、"废气"这些词的文档，而不会返回"脱硫工艺"、"尾气处理"等相关内容。你需要的是向量检索（Vector Search），而不是这个玩具引擎。
2. **上下文截断：** 你的 `Document.content` 字段存储的是完整文本，一个文档可能几千字，塞进 LLM 的上下文窗口，立刻超 Token。你需要在索引时对文本做分段（Chunking），而不是盲目拼接。
3. **响应延迟：** 当前 `call_llm` 是同步调用，RAG 需要检索 + 生成两步，如果都同步执行，API 响应时间会在 5-10 秒以上，用户体验极差。你需要异步处理 + 流式输出（Streaming）。

### 5.3 改造建议（如果一定要做）
1. **替换检索引擎：** 集成向量数据库（如 Qdrant、Milvus）或使用 MeiliSearch 的向量搜索功能（如果支持）。
2. **文本分段：** 在 `index_document` 时，将长文档切分成 500-1000 Token 的片段，每个片段单独索引。
3. **异步架构：** RAG 接口应该立即返回一个任务 ID，后台处理完成后通过 WebSocket 或轮询推送结果。

## 6. Linus 风格的改进命令

1. **重写爬虫状态管理：** 把那些全局变量和蹩脚的锁全部删掉，用真正的并发原语（如 `asyncio.Lock`）和状态机。
2. **重构 AI 调用：** 把 `categorize_article` 改成异步任务，用 Celery 或 RQ，爬虫只负责触发任务，不等待结果。
3. **清理 API 层：** `ai.py` 里的业务逻辑全部扔进 `services`，API 层只做参数校验和调用。
4. **修复搜索转义：** 写一个完整的 MeiliSearch 过滤器转义函数，或者用官方 SDK 的参数化查询。
5. **停止在 `main.py` 挂静态文件：** 生产环境用 Nginx，别让 FastAPI 干它不该干的活。

---

## 7. 最终评价
这项目就像一个刚学会写 Python 的大学生，用 GPT 生成的代码拼凑出来的原型。能跑，但每一步都踩在坑边上。如果你想把它当产品用，建议先花两个月时间重构，或者干脆重写。
